# WorldPainterWeb Compatibility Strategy

## Hard Requirements

The project now has four hard requirements:

- run entirely in the browser
- load original WorldPainter project files
- save project files in a form compatible with original WorldPainter
- export Minecraft worlds from the browser

These requirements are not secondary features. They define the architecture.

## Compatibility Target

The product target is broad behavior parity with the Java version of WorldPainter.

For implementation and validation purposes, the compatibility contract is semantic compatibility rather than byte-for-byte identity.

That means:

- a project created or edited in the browser should open in original WorldPainter where the represented feature set overlaps
- a project created in original WorldPainter should open in the browser where the represented feature set overlaps
- Minecraft world export from the browser should produce a usable world with equivalent terrain and supported layer behavior

In other words, semantic compatibility is the minimum technical bar. It is not meant to redefine the product vision downward into a permanently partial clone.

Not guaranteed:

- identical Java serialization bytes
- support for every plugin-defined serialized class on day one
- support for every exporter and every historical WorldPainter version on day one

## Recommended Technical Shape

Use a three-layer compatibility model:

1. internal canonical model in TypeScript
2. original WorldPainter file adapter
3. Minecraft export adapter

The internal model is the product core. The adapters are compatibility boundaries.

## Original WorldPainter File Strategy

The existing desktop project format is Java-centric. It should be treated as an external compatibility format, not as the internal runtime format.

Recommended path:

- parse metadata and project payload into a browser-native canonical model
- preserve unsupported or unknown fields where practical
- write a compatible project file from the canonical model

Important consequence:

- round-trip support needs explicit preservation rules for fields the browser editor does not understand yet

## Minecraft World Export Strategy

Minecraft world export should also be treated as an adapter layer.

Required building blocks:

- NBT reading and writing
- region and chunk writing
- block and material mapping
- supported layer export rules
- browser-safe packaging and download flow

The export path should be validated against golden worlds produced by original WorldPainter.

## Compatibility Levels

To avoid vague promises, each feature should be tagged with a support level:

- level 1: browser can read it
- level 2: browser can preserve it through save
- level 3: browser can edit it
- level 4: browser can export it into a Minecraft world

This should apply to:

- terrain heights
- water levels
- terrain materials
- supported layers
- world metadata
- dimension metadata
- custom objects
- plugin-provided data

## Test Corpus

A compatibility-first project needs fixture data early.

The repository should eventually maintain a corpus containing:

- small original .world files from multiple WorldPainter versions
- worlds using only core features
- worlds using advanced terrain and layers
- exported Minecraft worlds generated by original WorldPainter
- edge cases such as tall height ranges, custom dimensions, and large tile counts

Each fixture should have expected outcomes for:

- browser load
- browser save
- desktop reopen
- browser export

## Main Risks

### Java serialization compatibility

The original project format is expensive to reproduce safely in the browser.

Mitigation:

- target semantic compatibility
- preserve unknown payloads where practical
- maintain a feature compatibility matrix instead of claiming universal support

### Plugin and custom data

Third-party plugin data may not be interpretable in the browser runtime.

Mitigation:

- preserve opaque data where possible
- clearly mark plugin compatibility as partial until implemented

### Export parity

Minecraft export is large enough to become its own subsystem.

Mitigation:

- start with a constrained export target and explicit version support
- compare exported results against reference outputs from original WorldPainter

## Immediate Engineering Rule

No internal data model decision should be accepted unless it can plausibly support:

- import from original .world
- save back to a compatible .world
- export to Minecraft world

If a proposed shortcut makes one of those impossible, it conflicts with the hard requirements.
